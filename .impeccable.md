## Design Context

### Users

Hackathon jurors, security-minded engineers, and developers evaluating how to run autonomous agents safely on Linux. They need to understand the product boundary quickly, inspect the technical proof, and decide whether the approach is worth adopting after the demo.

### Brand Personality

Calm, rigorous, and operational. The interface should create confidence without pretending that a benchmark or a sandbox removes every side effect. It should feel like a trustworthy control room for a reversible transaction.

### Aesthetic Direction

Editorial technical-lab: warm paper surfaces, ink-like navy, and a single safety-orange accent. Use a restrained grid with a few deliberate instrument-panel details (status rails, evidence labels, measured bars) rather than generic dashboard cards. Light-first, with strong contrast and reduced-motion support. Avoid neon cyberpunk, generic SaaS gradients, and dark-mode-only terminal aesthetics.

### Design Principles

1. Lead with the invariant: the lower layer stays untouched while the agent works in a disposable upper layer.
2. Separate measured evidence from roadmap promises and external competitor claims.
3. Use progressive disclosure: a juror can scan the story, while an engineer can expand into commands, policy semantics, and benchmark caveats.
4. Make state legible through language, labels, and structure—not color alone.
5. Keep the implementation modular and publishable as a dependency-free static site.
